# RiSC-16 OS & Interrupt Architecture

The RiSC-16 handles interaction between user programs and the Kernel via two mechanisms: **Software Traps** and *
*Hardware Interrupts**.

## 1. Software Traps (Syscalls)

Triggered by the instruction `JALR rA, rB, imm7` where `imm7 != 0`.

- **The Vector ID**: The value of `imm7` (bits 6-0) tells the OS which service is requested.
- **The Mechanism**:
    1. The CPU identifies `imm7 > 0`.
    2. The CPU pauses internal execution and triggers the `onSyscall(cpu, vectorId)` Kotlin callback.
    3. Register usage convention for Syscalls:
        - **R1**: Argument 1 / String Pointer.
        - **R2**: Argument 2 / Return Value.
        - **R3**: Argument 3.

## 2. Hardware Interrupts (Preemption)

Triggered by external events (Timer, Keyboard, Network).

- **The Mechanism**:
    1. The Kernel sets a `pendingInterrupt` flag on the CPU.
    2. Before the next Fetch cycle, the CPU checks this flag.
    3. If true, the CPU:
        - Saves current `PC` into **R7** (The Link Register).
        - Forces `PC` to **0x0001** (Hardcoded Interrupt Vector).
        - Resumes execution (now inside the Kernel).

## 3. Kernel Entry Points (Vector Table)

The first few words of RAM are reserved for the Kernel:

- **0x0000**: Reset Vector (Where the CPU starts on boot).
- **0x0001**: Hardware Interrupt Entry Point.
- **0x0002**: Exception Entry Point (Division by zero, Segfault).
